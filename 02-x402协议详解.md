# x402 协议详解 — AI 代理间的加密货币支付

> 本文详细介绍 x402 协议（Coinbase）和 Google 的 A2A x402 Extension，包括技术架构、支付流程、安全设计和两者的区别。

- **来源**: [GitHub - google-agentic-commerce/a2a-x402](https://github.com/google-agentic-commerce/a2a-x402)
- **底层协议**: [x402 Protocol](https://x402.gitbook.io/x402)（由 Coinbase 和 Cloudflare 开发）
- **版本**: v0.1（2025-09-16 发布）
- **许可**: Apache-2.0

## 1 一句话理解

> AP2 解决的是"用户让 AI 代理帮自己去商家那里买东西"的支付问题。
> **A2A x402 解决的是另一个问题："AI 代理 A 请 AI 代理 B 帮忙干活，A 怎么付钱给 B？"**
>
> 就像你（用户）不需要参与，两个 AI 代理自己谈好价格、自己用加密货币结算。

## 2 它解决什么问题？

在 AI 代理生态中，一个代理可能需要调用另一个代理的能力（比如生成图片、翻译文本、分析数据）。问题是：**这些服务怎么收费？**

传统方式：注册账号 → 申请 API Key → 绑定信用卡 → 按月结算。这对 AI 代理来说太复杂了。

x402 的方式：**即用即付，无需注册，加密货币实时结算**。

## 3 x402 底层协议

A2A x402 是基于 **x402 协议** 构建的。x402 协议复活了 HTTP 状态码中从未被广泛使用的 **402 Payment Required**。

核心思想非常简单：

```
传统 HTTP 请求：
  客户端 → 请求资源 → 服务器返回 200 OK + 数据

x402 HTTP 请求：
  客户端 → 请求资源 → 服务器返回 402 Payment Required（告诉你要付多少钱）
  客户端 → 签名支付 → 重新请求（附带支付凭证）
  服务器 → 验证支付 → 链上结算 → 返回 200 OK + 数据
```

| 特性 | 说明 |
|------|------|
| **结算速度** | 约 2 秒 |
| **最小金额** | 低至 $0.01（适合微支付） |
| **默认代币** | USDC（美元稳定币） |
| **支持的链** | Base、Ethereum、Polygon 等 |
| **平台费用** | 零平台费，商户直收 |

## 4 A2A x402 的完整支付流程

在 A2A（Agent-to-Agent）协议中，x402 定义了四个角色：

| 角色 | 说明 | 类比 |
|------|------|------|
| **Client Agent** | 需要服务的 AI 代理 | 雇主 |
| **Merchant Agent** | 提供付费服务的 AI 代理 | 服务商 |
| **Signing Service** | 管理钱包私钥、负责签名 | 出纳/财务 |
| **Facilitator** | 验证支付并在链上结算 | 银行/清算所 |

完整交易流程（以"AI 代理 A 请 AI 代理 B 生成一张图片"为例）：

```
  Client Agent (A)                    Merchant Agent (B)               区块链
       │                                     │                          │
  ① A: "帮我生成一张猫的图片"                  │                          │
       │ ─── 发送请求 ──────────────────────→ │                          │
       │                                     │                          │
       │    B: "可以，但要付 0.05 USDC"        │                          │
       │ ←── 返回 402 + 支付要求 ──────────── │                          │
       │                                     │                          │
       │   支付要求包含：                       │                          │
       │   · 金额：0.05 USDC                  │                          │
       │   · 收款地址：0xB的钱包                │                          │
       │   · 支持的链：Base                    │                          │
       │   · 有效期：600秒                     │                          │
       │                                     │                          │
  ② A 决定接受，用钱包签名支付授权               │                          │
       │ ─── 发送签名后的支付凭证 ───────────→ │                          │
       │                                     │                          │
       │                               ③ B 验证签名                     │
       │                                  链上结算 ─────────────────────→│ 转账
       │                                     │                          │ 0.05 USDC
       │                                     │                          │ A → B
       │                               ④ B 开始生成图片                  │
       │                                     │                          │
       │    B: "搞定了，这是你的图片"           │                          │
       │ ←── 返回结果 + 链上交易凭证 ───────── │                          │
       │                                     │                          │
       │   凭证包含：                          │                          │
       │   · 交易哈希：0xabc123...             │                          │
       │   · 网络：Base                       │                          │
       │   · 状态：成功                        │
```

## 5 支付状态机

交易有 6 个状态，流转清晰：

```
                    ┌──────────────────────┐
                    │   payment-required   │  商户发出支付要求
                    └──────────┬───────────┘
                               │
                ┌──────────────┼──────────────┐
                ↓                             ↓
   ┌────────────────────┐        ┌───────────────────┐
   │  payment-rejected  │        │ payment-submitted  │  客户提交签名支付
   └────────────────────┘        └─────────┬─────────┘
         (客户拒绝)                         ↓
                              ┌─────────────────────┐
                              │  payment-verified    │  商户验证签名通过
                              └─────────┬───────────┘
                                        │
                         ┌──────────────┼──────────────┐
                         ↓                             ↓
              ┌─────────────────┐          ┌──────────────────┐
              │ payment-completed│          │  payment-failed   │
              └─────────────────┘          └──────────────────┘
                (链上结算成功)                (结算失败)
```

## 6 安全设计（含原版 x402 与 A2A x402 的密钥管理对比）

| 安全措施 | 说明 |
|---------|------|
| **私钥隔离** | 私钥绝对不能交给 LLM 处理，必须由独立的签名服务管理 |
| **签名验证** | 商户必须在结算前密码学验证每一笔支付签名 |
| **防重放攻击** | 服务端必须追踪已使用的 nonce，防止同一笔支付被重复提交 |
| **传输加密** | 所有通信必须使用 HTTPS/TLS |

### 原版 x402 的密钥安全方案

原版 x402 协议本身**不规定密钥如何存储**，只给了一条原则和一个推荐方案：

**原则**：买方在本地签名，卖方只验证签名、永远不持有买方的私钥。

**推荐方案**：使用 Coinbase 的 **CDP Wallet API**（Server Wallet v2）来管理密钥。

CDP Wallet API 底层使用 **AWS Nitro Enclaves（可信执行环境 TEE）**，核心安全设计如下：

```
┌─────────────────────────────────────────────────────┐
│              AWS Nitro Enclave (TEE)                 │
│                                                     │
│  · 私钥在 TEE 内部生成                                │
│  · 私钥在 TEE 内部加密                                │
│  · 签名操作在 TEE 内部完成                             │
│  · 私钥永远不离开这个隔离环境                           │
│                                                     │
│  特性：                                              │
│  · 无持久化存储（关机即清除）                           │
│  · 无交互式访问（没人能"登录"进去看）                   │
│  · 无外部网络连接（只接受签名请求，不主动联网）           │
│                                                     │
│  加密后的密钥版本存储在 CDP 数据库中                    │
│  → 必须配合 Wallet Secret 才能解密使用                  │
│  → Wallet Secret 使用 ECDSA secp256r1 椭圆曲线认证     │
└─────────────────────────────────────────────────────┘
```

用生活类比：TEE 就像一个**透明的保险柜**——你可以把需要签名的文件递进去，里面的机器人帮你签好再递出来，但没人能打开保险柜、拿走里面的印章。

**但原版 x402 有一个关键缺陷**：它只是**建议**你用安全钱包，并没有在协议层面做强制分离。也就是说：

```
原版 x402 允许这种写法（不安全但协议不阻止）：

  const wallet = new Wallet(privateKey);  // 私钥直接写在代码里
  const signature = wallet.sign(payload);  // 程序直接签名
```

对于普通的程序调 API 场景，这个风险还可控（开发者自己负责）。
但对于 AI 代理场景，就很危险了——LLM 可能被 prompt injection 攻击，诱导它泄露或滥用私钥。

### Google A2A x402 的安全升级

Google 的 A2A x402 Extension 在协议层面**强制要求**架构分离：

```
原版 x402（建议但不强制）：
  ┌───────────────────────┐
  │     Client 程序        │
  │  · 业务逻辑            │
  │  · 钱包/私钥管理  ←危险 │
  │  · 签名操作            │
  └───────────────────────┘

A2A x402（协议层面强制）：
  ┌───────────────────┐          ┌───────────────────┐
  │  Client Agent      │          │  Signing Service   │
  │  (LLM 驱动)        │ ──请求──→│  (独立服务)         │
  │                    │          │                    │
  │  · 业务逻辑         │          │  · 管理私钥         │
  │  · 决定是否付款      │←─签名──  │  · 执行签名         │
  │  · ❌ 碰不到私钥    │          │  · 验证权限         │
  └───────────────────┘          └───────────────────┘
        ↑                               ↑
   LLM 可能被攻击               私钥在这里，与 LLM 物理隔离
   但拿不到私钥
```

| 对比 | 原版 x402 | A2A x402 Extension |
|------|-----------|-------------------|
| 密钥存储规范 | 协议不规定，建议用 CDP Wallet API | 协议强制要求与 LLM 分离 |
| 安全级别 | 取决于开发者实现 | 架构层面保证 |
| 推荐方案 | Coinbase CDP (TEE) | 独立 Signing Service |
| 防 prompt injection | 未考虑（设计时没有 LLM 场景） | 核心设计目标 |
| 适用场景 | 普通程序/脚本调用 API | AI 代理（LLM 驱动）|

## 7 与 AP2 的关系

| 维度 | AP2 | A2A x402 |
|------|-----|----------|
| **场景** | 用户 → AI 代理 → 商家（帮用户买东西） | AI 代理 A → AI 代理 B（代理之间买服务） |
| **支付方式** | 信用卡、借记卡、稳定币等（全支持） | 仅加密货币（USDC 为主） |
| **用户参与** | 需要用户签名授权 | 代理自主完成，用户可能不知情 |
| **结算方式** | 走传统支付网络（Visa/Mastercard） | 链上结算（Base/Ethereum/Polygon） |
| **典型金额** | 日常购物（几十到几千元） | 微支付（$0.01 起） |
| **关联** | AP2 通过 x402 扩展支持加密货币支付 | x402 是 AP2 在加密支付场景的具体实现 |

简单说：**AP2 是面向消费者购物的协议，A2A x402 是面向代理之间服务交易的协议**，两者通过 x402 扩展打通。

## 8 Google 的 A2A x402 到底在原版 x402 上扩展了什么？

### 先理解原版 x402（Coinbase 开发）

原版 x402 是一个**非常简洁的 HTTP 支付协议**，设计理念是"像访问网页一样付款"：

```
原版 x402 流程（纯 HTTP）：

客户端 ──── GET /api/generate-image ──────────────→ 服务器
客户端 ←─── 402 + PAYMENT-REQUIRED 头（JSON）──── 服务器     "要付 0.05 USDC"
客户端 ──── GET /api/generate-image ──────────────→ 服务器     重发请求
             + PAYMENT-SIGNATURE 头（签名后的支付）
客户端 ←─── 200 OK + 图片数据 ────────────────── 服务器
             + PAYMENT-RESPONSE 头（链上交易凭证）
```

核心特点：**无状态、一问一答、通过 HTTP 头传递支付信息、没有"代理"的概念**。

### Google 扩展了什么？（6 大增强）

**增强 1：从"无状态 HTTP"升级为"有状态任务"**

| 原版 x402 | A2A x402 Extension |
|-----------|-------------------|
| 每次请求独立，无状态 | 引入 `taskId` 关联整个交易生命周期 |
| 支付和服务是一次 HTTP 往返 | 支付和服务可以是多步异步流程 |

原版 x402 适合简单的"付钱→拿结果"场景。但 AI 代理的服务可能很复杂（比如生成视频需要 10 分钟），需要跟踪状态。Google 用 A2A 的 **Task 对象** 解决了这个问题——一个 taskId 串起整个"请求→付款→处理中→完成"的过程。

**增强 2：定义了明确的支付状态机**

原版 x402 没有定义支付状态，只有 HTTP 状态码（402 / 200）。Google 定义了 **6 个显式状态**：

```
payment-required → payment-submitted → payment-verified → payment-completed
                 ↘ payment-rejected                     ↘ payment-failed
```

每个状态都写在消息的 metadata 里，任何参与方都能精确知道"当前这笔交易到哪一步了"。

**增强 3：支付信息从 HTTP 头搬到了消息体**

| 原版 x402 | A2A x402 Extension |
|-----------|-------------------|
| 支付要求放在 HTTP `PAYMENT-REQUIRED` 头 | 支付要求放在 A2A Message 的 `metadata` 字段 |
| 支付签名放在 HTTP `PAYMENT-SIGNATURE` 头 | 支付签名放在 A2A Message 的 `metadata` 字段 |
| 结算凭证放在 HTTP `PAYMENT-RESPONSE` 头 | 结算凭证放在 A2A Message 的 `metadata` 字段 |

为什么要这样做？因为 A2A 代理之间的通信不一定是简单的 HTTP 请求-响应。它们通过 A2A 协议的 `message/send` 方法交换结构化消息，把支付信息嵌入消息体更自然、更灵活。

**增强 4：代理能力发现机制**

原版 x402：客户端必须**事先知道**某个 URL 需要付费，去请求后才收到 402。

A2A x402：商户代理通过 **AgentCard**（代理名片）提前声明"我支持 x402 支付"：

```json
{
  "capabilities": {
    "extensions": [{
      "uri": "https://github.com/google-a2a/a2a-x402/v0.1",
      "description": "支持 x402 链上支付",
      "required": true
    }]
  }
}
```

客户端代理在决定调用某个服务之前，就能通过 AgentCard 知道"这个代理要收费、用什么方式收费"，而不是盲目请求后才发现要付钱。

**增强 5：强制分离签名服务（安全增强）**

原版 x402：只说"客户端用钱包签名"，没有规定钱包和客户端的关系。

A2A x402 **明确要求**：签名私钥必须由独立的 **Signing Service** 管理，**绝对不能交给 LLM**。

```
原版 x402：
  客户端（含钱包）→ 签名 → 发送

A2A x402：
  Client Agent (LLM) → 请求签名 → Signing Service (独立服务，管理私钥) → 返回签名
                       ↑ LLM 永远碰不到私钥
```

这是因为 LLM 可能被 prompt injection 攻击，如果 LLM 能直接操作私钥，攻击者可以诱导它转走所有资金。

**增强 6：交易凭证持久化记录**

原版 x402：结算凭证在 HTTP 响应头返回一次就结束了。

A2A x402：引入 `x402.payment.receipts` —— 一个**持久化数组**，记录该 Task 生命周期内所有的结算尝试（包括失败的）：

```json
"x402.payment.receipts": [
  { "success": false, "errorReason": "insufficient_funds", "network": "base" },
  { "success": true, "transaction": "0xabc123...", "network": "base", "payer": "0x..." }
]
```

这意味着如果第一次付款失败（余额不足），代理换个钱包重试成功，整个过程都有完整记录，方便对账和审计。

### 一图总结

```
┌──────────────────────────────────────────────────────────────┐
│                    原版 x402 (Coinbase)                       │
│                                                              │
│  · 纯 HTTP 协议（402 状态码 + HTTP 头）                        │
│  · 无状态，一问一答                                            │
│  · 没有代理/任务概念                                           │
│  · 客户端自己管钱包                                            │
│  · 单次凭证返回                                               │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│              Google A2A x402 Extension 新增                   │
│                                                              │
│  ✦ Task 状态管理（taskId 关联完整交易生命周期）                   │
│  ✦ 6 状态支付状态机（精确追踪每一步）                            │
│  ✦ 支付信息嵌入 A2A Message metadata（更结构化）                │
│  ✦ AgentCard 能力发现（提前知道要付费）                         │
│  ✦ 强制签名服务分离（LLM 不碰私钥）                            │
│  ✦ 交易凭证持久化数组（完整审计记录）                            │
│  ✦ 7 种标准错误码（统一错误处理）                               │
│                                                              │
│  本质：把 x402 的支付能力"嵌入"到 A2A 代理通信框架中             │
└──────────────────────────────────────────────────────────────┘
```

简单说：**原版 x402 是给"程序调用 API"设计的支付协议，Google 把它改造成了给"AI 代理之间协作"设计的支付协议**。核心支付逻辑没变（还是用加密货币链上结算），但套上了 A2A 的代理通信框架，加了状态管理、能力发现、安全隔离和审计追踪。
