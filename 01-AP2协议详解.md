# AP2 协议详解 — Google Agent Payments Protocol

> 本文详细介绍 Google 的 AP2（Agent Payments Protocol）协议的核心机制、技术架构和生态。

---

- **来源**: [Google Cloud Blog - Announcing AP2](https://cloud.google.com/blog/products/ai-machine-learning/announcing-agents-to-payments-ap2-protocol)
- **发布日期**: 2025-09-16
- **发布方**: Google（Stavan Parikh, VP/GM Payments; Rao Surapaneni, VP/GM Business Applications Platform）

## 1 概述

AP2 是 Google 联合支付和科技公司开发的**开放协议**，用于在跨平台环境下安全地发起和完成 AI 代理主导的支付交易。该协议可作为 **Agent2Agent (A2A)** 协议和 **Model Context Protocol (MCP)** 的扩展，建立一个与支付方式无关的框架，让用户、商户和支付提供商能够放心地使用各类支付方式进行交易。

## 2 解决的核心问题

现有支付体系的基本假设是"人类直接点击购买"，AI 代理自主发起支付打破了这一假设，AP2 针对性解决三个关键问题：

| 问题 | 说明 |
|------|------|
| **授权 (Authorization)** | 证明用户授予了代理进行特定购买的权限 |
| **真实性 (Authenticity)** | 确保代理的请求准确反映用户的真实意图 |
| **问责 (Accountability)** | 当欺诈或错误交易发生时，明确责任归属 |

## 3 核心机制 —— Mandate（授权凭证）

#### 一句话理解

> 传统支付：你亲自走进商店，挑好商品，亲手刷卡 —— 商家看到你本人，所以信任这笔交易。
> AI 代理支付：你派了一个"助理"去商店帮你买东西 —— 商家怎么确认这个助理真的是你派来的、真的按你的意思买的？
> **Mandate 就是你给这个助理的"授权书"**，上面有你的签名、明确的购买规则，任何人都可以验证它的真实性，且无法伪造或篡改。

#### 什么是 Mandate？

Mandate 是一份**加密签名的数字合约**，具备以下特性：

- **防篡改**：一旦签署，内容无法被任何人（包括 AI 代理自己）修改
- **可验证**：商家、支付机构等任何参与方都能独立验证其真实性
- **可追溯**：形成完整的证据链，出了问题可以回溯追责

技术上，Mandate 由 **Verifiable Credentials (VCs，可验证凭证)** 签名。可以类比为：传统世界中经过公证的委托书，但在数字世界中通过密码学实现，更快、更安全。

#### 两种 Mandate 类型

| 类型 | 用途 | 类比 |
|------|------|------|
| **Intent Mandate（意图授权）** | 记录用户"想买什么"的意图和规则 | 你写给助理的购物清单 + 预算限制 |
| **Cart Mandate（购物车授权）** | 锁定最终要买的具体商品和价格 | 你在订单上签字确认"就买这些" |

#### 场景一：实时购买（你在旁边盯着）

就像你在手机上让 AI 助手帮你买东西，你全程参与：

```
第1步：你说"帮我找一双白色跑鞋"
       → 系统生成【Intent Mandate】记录你的意图
       → 这份记录有你的加密签名，证明确实是你提出的请求

第2步：AI 代理去各个商家比价，找到合适的鞋子，呈现给你
       → 你看到：Nike Air Max, 白色, ¥899

第3步：你确认"就买这双"
       → 系统生成【Cart Mandate】锁定具体商品和价格
       → 这份记录同样有你的加密签名
       → 此时商品和价格被"冻结"，不能被偷偷更改

第4步：Cart Mandate 关联你的支付方式（信用卡/支付宝等）→ 完成付款
```

**关键点**：每一步都有你的签名确认，商家能验证"确实是用户本人授权买的这双鞋"。

#### 场景二：委托任务（你不在，代理自主行动）

就像你提前告诉 AI 助手一个任务，然后去忙别的：

```
第1步：你说"帮我抢周杰伦演唱会门票，最多花 ¥2000，只要内场"
       → 系统生成【Intent Mandate】，内容包含：
         · 商品：周杰伦演唱会门票
         · 价格上限：¥2000
         · 条件：仅限内场座位
         · 时间窗口：开售后 30 分钟内
       → 你用加密签名确认这份授权

第2步：AI 代理持续监控票务平台...

第3步：开售！代理发现内场票 ¥1800，符合所有条件
       → 代理依据 Intent Mandate 的规则，自动生成【Cart Mandate】
       → Cart Mandate 里写明：1张内场票, ¥1800
       → 因为完全符合你预设的规则，代理有权自动执行

第4步：Cart Mandate 关联支付方式 → 自动完成付款
```

**关键点**：代理不是"随便买"，而是严格按照你预设的规则执行。如果票价超过 ¥2000 或者不是内场，代理**无权**生成 Cart Mandate，交易不会发生。

#### 信任链条（为什么这套机制能防止出问题）

```
 用户签名的 Intent Mandate          用户/代理签名的 Cart Mandate           支付凭证
 ┌─────────────────────┐          ┌─────────────────────────┐          ┌──────────┐
 │ "我想买什么"         │ ──────→ │ "确认买这个，这个价格"    │ ──────→ │ 扣款记录  │
 │ + 规则/预算/条件     │          │ + 具体商品和金额         │          │          │
 │ + 用户加密签名       │          │ + 加密签名               │          │          │
 └─────────────────────┘          └─────────────────────────┘          └──────────┘
         ↑                                  ↑                               ↑
    证明用户授权了               证明买的东西符合用户意图            证明钱付对了
```

这三个环节串在一起，形成一条**不可否认的审计链**：
- **出了纠纷**：调出 Mandate 链条，一目了然是用户的意图、代理的执行还是商家的问题
- **防止代理越权**：代理只能在 Intent Mandate 规定的范围内操作
- **防止篡改**：所有环节都有加密签名，改一个字节都会导致验证失败

## 4 Mandate 的技术本质与存储方式

> **常见疑问**：Mandate 存在区块链上吗？

#### 结论：不在区块链上

AP2 官方将 Mandate 称为 **Verifiable Digital Credentials (VDCs，可验证数字凭证)**。它本质上是一段**加密签名的结构化数据**（JSON 格式），而非区块链上的记录。

#### 它到底是什么？

可以把 VDC 想象成一份**带有防伪印章的电子文档**：

```
┌─────────────────────────────────────────────┐
│          一个 Cart Mandate (VDC) 的内容        │
├─────────────────────────────────────────────┤
│  · 买家身份标识（加密的用户 ID）               │
│  · 卖家身份标识（商户 ID）                     │
│  · 交易详情（商品、数量、金额、币种）            │
│  · 支付方式的 Token 化表示（不暴露卡号）         │
│  · 风险相关信号（设备信息、位置等）              │
│  · 时间戳                                     │
│  ─────────────────────────────────────────── │
│  🔏 用户的加密签名（用户私钥生成）              │
│  🔏 商户的加密签名                             │
└─────────────────────────────────────────────┘
```

这里的"加密签名"使用的是**非对称加密**（公钥/私钥体系），和区块链上的签名技术同源，但不依赖区块链网络。

#### 存储在哪？谁持有？

AP2 采用**角色分离架构**，不同的 VDC 由不同的参与方持有和传递：

| 角色 | 说明 | 持有的 VDC |
|------|------|-----------|
| **User Agent (UA)** | 用户的 AI 代理（如 Gemini、ChatGPT） | Intent Mandate |
| **Credentials Provider (CP)** | 管理用户支付凭证的实体（如数字钱包） | 支付凭证的 Token |
| **Merchant Endpoint (ME)** | 商户的代理/接口 | Cart Mandate（由商户生成，用户签名） |
| **Merchant Payment Processor (MPP)** | 商户的支付处理方 | Payment Mandate → 发给支付网络 |
| **Network & Issuer** | 支付网络和发卡行（Visa/Mastercard等） | Payment Mandate（用于风控判断） |

流转过程：

```
用户设备                商户代理              支付处理方           支付网络
  │                      │                    │                   │
  │  ① 用户签署          │                    │                   │
  │  Intent Mandate      │                    │                   │
  │  (存在用户设备/代理)   │                    │                   │
  │                      │                    │                   │
  │ ──── 购物意图 ──────→ │                    │                   │
  │                      │                    │                   │
  │ ←── 生成购物车 ────── │                    │                   │
  │                      │                    │                   │
  │  ② 用户签署          │                    │                   │
  │  Cart Mandate        │                    │                   │
  │ ──── 签名后传给商户 →  │                    │                   │
  │                      │                    │                   │
  │                      │ ── Cart Mandate ──→ │                   │
  │                      │                    │                   │
  │                      │                    │ ③ 生成             │
  │                      │                    │ Payment Mandate    │
  │                      │                    │ ── 发送给网络 ────→ │
  │                      │                    │                   │
  │                      │                    │                   │ ④ 风控验证
  │                      │                    │                   │ 批准/拒绝
```

#### 为什么不用区块链？

| 维度 | 区块链方案 | AP2 的 VDC 方案 |
|------|-----------|----------------|
| **速度** | 上链确认需要数秒到数分钟 | 直接在参与方之间传递，毫秒级 |
| **成本** | 每笔交易需要 Gas 费 | 无链上成本 |
| **隐私** | 链上数据公开可查 | 数据仅在必要的参与方之间共享 |
| **兼容性** | 需要所有参与方接入同一条链 | 兼容现有支付体系（Visa/Mastercard等） |
| **防篡改** | 靠区块链共识机制 | 靠密码学签名（同样不可篡改） |

简单说：AP2 借用了区块链的**密码学思想**（加密签名、不可篡改），但没有用区块链的**基础设施**（分布式账本、共识机制）。这是一个务实的选择——让协议能无缝嵌入现有的 Visa/Mastercard 支付网络，而不需要商家和银行去跑区块链节点。

> **例外**：当支付方式本身是加密货币/稳定币时（通过 x402 扩展），结算环节会涉及链上操作，但 Mandate 本身的授权和验证仍然是链下的。

## 5 支持的支付方式

AP2 设计为通用协议，支持多种支付类型：

- 信用卡 / 借记卡
- 稳定币 (Stablecoins)
- 加密货币 (Cryptocurrencies)
- 实时银行转账

其中，Google 与 Coinbase、以太坊基金会、MetaMask 等合作推出了 **A2A x402 扩展**，专门支持基于加密货币的代理支付。

## 6 应用场景示例

| 场景 | 描述 |
|------|------|
| **智能购物** | 用户想要某款绿色夹克但缺货，代理持续监控库存和价格，在条件满足时自动购买 |
| **个性化推荐** | 用户告知代理想买自行车用于旅行，商家代理据此生成包含头盔、行李架的限时捆绑优惠 |
| **协调任务** | 用户设定预算 $700 预订周末旅行，代理同时与航空公司和酒店代理交互完成组合预订 |

## 8 关键要点

1. AP2 是**开放协议**，旨在防止生态碎片化
2. 基于 **A2A + MCP** 扩展，不是独立体系
3. 核心创新是 **VDC/Mandate 机制**（加密签名的授权凭证链，链下传递，不依赖区块链）
4. **支付方式无关**，同时支持传统支付和加密支付
5. 已有 **60+ 机构**参与，覆盖支付全链条
6. 代码和规范开源在 [GitHub (google-agentic-commerce/AP2)](https://github.com/google-agentic-commerce/AP2)
7. 官方文档：[AP2 Documentation](https://google-agentic-commerce.github.io/AP2/)
8. 未来方向：与标准组织合作（如 FIDO Alliance），推进去中心化身份和代理授权
