# AI 支付科普与协议全景

> 本文从零开始解释"AI 支付"是什么、为什么需要、有哪些协议和概念，以及它们之间的关系。
> 适合对 AI 代理支付领域不了解的人快速建立全景认知。

---

## 第一部分：什么是 AI 支付？

### 1. 从"人类支付"到"AI 支付"

过去 30 年，支付的进化路线是这样的：

```
现金 → 刷卡 → 网上支付 → 手机支付 → ???
```

下一步是什么？**AI 代理帮你付款，或者 AI 代理之间自己付款。**

这就是"AI 支付"——不再是人类亲自操作支付，而是 AI 代理（Agent）代替你、甚至完全自主地完成交易。

### 2. 三种 AI 支付场景

| 场景 | 描述 | 举例 |
|------|------|------|
| **场景 A：AI 帮你买** | 你告诉 AI 你想买什么，AI 代理去商家那里帮你完成整个购买流程 | "Gemini，帮我买一双白色跑鞋" |
| **场景 B：AI 自己买** | 你提前授权 AI 在满足条件时自主购买，你不在场 | "帮我抢周杰伦演唱会门票，最多 ¥2000" |
| **场景 C：AI 付钱给 AI** | 一个 AI 代理调用另一个 AI 代理的服务，自动用加密货币结算 | AI 代理 A 付 $0.01 给 AI 代理 B 调用翻译 API |

```
  场景 A：人在场                场景 B：人不在场              场景 C：纯机器
  ┌──────────────┐           ┌──────────────┐           ┌──────────────┐
  │ 用户 → AI → 商户 │         │ 用户预授权     │           │ AI-A → AI-B   │
  │ 用户全程参与   │           │ AI 自主执行    │           │ 无人参与       │
  │ 信用卡/支付宝  │           │ 预设条件触发   │           │ 加密货币       │
  └──────────────┘           └──────────────┘           └──────────────┘
       ↑                           ↑                          ↑
    传统支付方式                  需要授权机制                 需要链上结算
```

### 3. 为什么现有支付系统搞不定？

今天所有支付系统（Visa、支付宝、微信支付）都基于一个假设：**有一个人类在屏幕前点击"确认支付"。**

当 AI 代理发起支付时，这个假设崩塌了：

| 问题 | 传统支付 | AI 代理支付 |
|------|---------|-----------|
| 谁在付款？ | 人类，看得见、摸得着 | AI 代理，一串代码 |
| 怎么证明是本人？ | 人脸/指纹/密码 | ？？？ |
| 买错了怎么办？ | 找客服退款 | AI 幻觉买错了谁负责？ |
| 被盗刷怎么办？ | 银行冻结 + 调查 | AI 被 prompt injection 攻击了怎么办？ |
| 交易记录？ | 银行账单 | 哪些是人类授权的？哪些是 AI 自作主张？ |

**这就是为什么需要一系列新的协议和标准——解决"AI 代表人类花钱"和"AI 之间互相花钱"的信任问题。**

## 第二部分：基础协议层（先理解这两个，其他的才能看懂）

AI 支付协议不是凭空出现的，它们建立在更基础的协议之上：

```
┌─────────────────────────────────────────────────────┐
│                 AI 支付协议                           │
│   AP2、ACP、x402、TAP 等                              │
├─────────────────────────────────────────────────────┤
│                 AI 代理通信协议                        │
│   A2A（代理之间怎么对话）                              │
├─────────────────────────────────────────────────────┤
│               AI 工具连接 + 技能协议                    │
│   MCP（连接外部工具）+ Skills Protocol（学会操作方法）    │
├─────────────────────────────────────────────────────┤
│                 AI 代理开发框架                        │
│   ADK、LangChain、AutoGen 等                         │
└─────────────────────────────────────────────────────┘
```

### 4. MCP —— Model Context Protocol（模型上下文协议）

- **开发者**：Anthropic（Claude 的公司）
- **发布时间**：2024 年 11 月
- **一句话理解**：**AI 的"USB-C 接口"**

#### 解决什么问题？

AI 模型（如 ChatGPT、Claude、Gemini）本身只会"思考和说话"，不能读你的文件、查数据库、调 API。要让 AI 有"动手能力"，需要给它接上各种工具。

```
没有 MCP 之前：
  AI 要读 Google Drive？写个专门的集成代码
  AI 要查 PostgreSQL？再写个专门的集成代码
  AI 要调 Slack？又写个专门的集成代码
  → 每个 AI + 每个工具 = 一个定制集成 → 排列组合爆炸

有了 MCP 之后：
  每个工具只需要开发一个"MCP Server"
  每个 AI 只需要内置一个"MCP Client"
  → 任意 AI × 任意工具 = 即插即用
```

类比：在 USB-C 出现之前，每个手机品牌用不同的充电口。MCP 就是 AI 世界的 USB-C——统一了 AI 调用外部工具的接口。

#### 与支付的关系

MCP 本身不是支付协议，但支付工具（如 Stripe）可以做成 MCP Server，让任意 AI 代理通过 MCP 调用支付能力。**MCP 是"AI 能用工具"的前提，支付只是工具之一。**

---

### 5. Skills Protocol —— 技能协议（代理技能发现与执行协议）

- **发布时间**：2025 年 10 月
- **规范基础**：JSON-RPC 2.0 over HTTP
- **一句话理解**：**AI 的"菜谱库"**

#### 解决什么问题？

MCP 让 AI 能"连上各种厨房工具"（数据库、API、文件系统），但没解决另一个问题：**AI 怎么知道用这些工具该按什么步骤操作？**

```
MCP 解决的是：                Skills 解决的是：
  "能连哪些工具？"               "拿到工具后怎么干活？"
  → 连接能力（where/what）       → 操作方法（how）

类比：
  MCP = 给你一个装满工具的厨房      Skills = 给你一本菜谱集
  有了厨房但不会做菜 → 需要菜谱     有了菜谱但没有厨房 → 也做不了
  → 两者互补，不是竞争
```

#### 核心机制

Skills Protocol 只暴露 **8 个元工具（meta-tools）**给 AI 代理，代理通过这些工具自主发现和执行技能：

| 工具 | 作用 |
|------|------|
| `list_skills` | 列举所有可用技能 |
| `describe_skill` | 查看技能说明书（manifest + 文档） |
| `read_skill_file` | 读取技能目录中的文件 |
| `execute_skill` | 运行技能的入口函数 |
| `run_code` | 在沙箱中执行自定义代码 |
| `create_blob` / `read_blob` | 大数据存取（避免撑爆上下文窗口） |
| `load_skills_protocol_guide` | 加载引导说明 |

每个技能（Skill）是一个目录，包含 `skill.toml` 清单文件 + 代码/文档：

```
my-skill/
  skill.toml          ← 清单：名称、版本、权限、入口函数
  main.py             ← 代码逻辑
  README.md           ← 使用说明
```

#### 关键设计：渐进式加载

Skills 的一大亮点是 **"渐进式披露"（Progressive Disclosure）**——代理不需要一次性加载所有技能的完整内容，而是先看标题（~100 tokens），需要时再加载详情。这避免了上下文窗口被技能说明撑满。

```
第一步：list_skills → 看到 50 个技能名称（几百 tokens）
第二步：describe_skill("支付处理") → 加载这一个技能的说明（几百 tokens）
第三步：execute_skill("支付处理", args) → 在沙箱中安全执行
→ 按需加载，不浪费上下文窗口
```

#### MCP vs Skills 快速对比

| 维度 | MCP | Skills Protocol |
|------|-----|-----------------|
| **解决什么** | AI 连接外部系统 | AI 学习操作方法 |
| **类比** | USB-C 接口 | 菜谱库 |
| **架构** | Client-Server 模型 | 文件系统 + JSON-RPC |
| **内容** | 工具声明 + 调用 | 指令 + 代码 + 文档 |
| **执行** | 远程服务端执行 | 本地沙箱隔离执行 |
| **关系** | 互补，MCP 提供能力 | 互补，Skills 提供方法 |

#### 与支付的关系

Skills Protocol 不是支付协议，但可以把"如何发起一笔 AP2 支付""如何处理 x402 的 402 响应"等复杂多步骤流程封装成 Skill。代理调用 MCP 连接 Stripe API，同时调用 Skill 学习完整的支付处理流程。**MCP 给工具，Skills 给方法，组合起来代理才能"独立干活"。**

### 6. A2A —— Agent2Agent Protocol（代理间通信协议）

- **开发者**：Google
- **发布时间**：2025 年 4 月
- **一句话理解**：**AI 代理之间的"通用语言"**

#### 解决什么问题？

MCP 解决了"AI 怎么用工具"，但没解决"AI 怎么和另一个 AI 对话协作"。

```
没有 A2A 之前：
  Google 的 AI → 定制接口 → Salesforce 的 AI
  OpenAI 的 AI → 另一套接口 → ServiceNow 的 AI
  → 每对 AI 之间都要定制通信方式

有了 A2A 之后：
  任意 AI 代理 ← A2A 协议 → 任意 AI 代理
  → 标准化通信，即插即用
```

#### 核心概念

| 概念 | 作用 | 类比 |
|------|------|------|
| **Agent Card** | 代理的"名片"，声明自己能做什么、怎么联系 | LinkedIn 个人主页 |
| **Task** | 一次协作的工作单元，有生命周期（提交→处理中→完成） | 工单系统的一个工单 |
| **Message** | 代理之间交换的消息（包含文本、文件、结构化数据） | 微信消息 |
| **Part** | 消息的组成部分（文本、图片、JSON 等） | 消息中的附件 |

#### 与支付的关系

A2A 是 AI 代理协作的通信框架。AP2 和 x402 的支付信息就是通过 A2A 的 Message 来传递的。**A2A 是"路"，支付数据是路上跑的"车"。**

---

### 7. ADK —— Agent Development Kit（代理开发工具包）

- **开发者**：Google
- **一句话理解**：**开发 AI 代理的"脚手架"**

ADK 是 Google 提供的 AI 代理开发框架，包含代理构建、工具集成、会话管理、评估、部署等能力。开发者用 ADK 构建代理 → 用 MCP 连接工具 → 用 Skills 学会操作方法 → 用 A2A 和其他代理通信 → 用 AP2/x402 完成支付。

```
ADK 在整个体系中的位置：

  开发者用 ADK 造代理
       ↓
  代理用 MCP 连接工具（数据库、API、文件系统等）
  代理用 Skills Protocol 学会操作方法（流程、步骤、最佳实践）
       ↓
  代理用 A2A 和其他代理对话协作
       ↓
  代理用 AP2/ACP/x402 完成支付
```

## 第三部分：AI 支付协议详解

现在基础概念清楚了，来看每个支付协议具体解决什么问题。

### 8. AP2 —— Agent Payments Protocol（代理支付协议）

| 属性 | 信息 |
|------|------|
| **开发者** | Google + 60 家合作方 |
| **发布时间** | 2025 年 9 月 |
| **解决场景** | 场景 A + B：AI 代理帮用户在商户购物 |
| **支付方式** | 全部（信用卡、借记卡、银行转账、稳定币、加密货币） |
| **核心创新** | Mandate（授权凭证）—— 加密签名的用户意图证明 |

#### 核心机制

AP2 的核心问题是：**商家怎么知道这个 AI 代理真的是用户派来的、真的按用户意思买的？**

解决方案是 **Mandate（授权凭证）**，一种加密签名的数字合约：

```
Intent Mandate（意图授权）："我想买白色跑鞋，预算 ¥1000"  + 用户签名
      ↓
Cart Mandate（购物车授权）："确认买这双 Nike ¥899"  + 用户签名
      ↓
Payment Mandate（支付授权）：发给银行/支付网络进行扣款
```

每一步都有用户的加密签名，形成不可篡改的证据链。

---

### 9. ACP —— Agentic Commerce Protocol（智能体商务协议）

| 属性 | 信息 |
|------|------|
| **开发者** | Stripe + OpenAI |
| **发布时间** | 2025 年 9 月 |
| **解决场景** | 场景 A：AI 代理帮用户在商户购物 |
| **支付方式** | 传统支付（信用卡、借记卡、40+ 种方式） |
| **核心创新** | SharedPaymentToken（SPT）—— 受限支付凭证 |

#### 核心机制

ACP 的思路不同于 AP2——它不用密码学签名，而是靠 **Stripe 平台**作为信任中心：

```
用户的信用卡存在 ChatGPT 上
      ↓
ChatGPT 生成一个 SPT（SharedPaymentToken）
  · 只能指定商户用
  · 限定金额上限
  · 限定有效期
  · 不包含真实卡号
      ↓
SPT 发给商户 → 商户通过 Stripe 扣款
```

本质区别：AP2 靠密码学证明"用户真的授权了"，ACP 靠 Stripe 平台信任"这个 Token 是合法的"。

### 10. x402 —— HTTP 402 支付协议

| 属性 | 信息 |
|------|------|
| **开发者** | Coinbase（+ Cloudflare 联合创建 x402 基金会） |
| **发布时间** | 2025 年 5 月 |
| **解决场景** | 场景 C：机器付钱给机器 |
| **支付方式** | 仅加密货币（USDC 为主） |
| **核心创新** | 复活 HTTP 402 状态码，让支付像 HTTP 请求一样简单 |

#### 核心机制

x402 的设计哲学是极简——把支付嵌入 HTTP 协议本身：

```
传统调用付费 API：
  注册账号 → 申请 API Key → 绑信用卡 → 按月结算
  → 复杂，AI 代理做不了

x402 调用付费 API：
  GET /api/data → 收到 402 "要付 $0.01 USDC"
  签名支付 → 重发请求 + 支付凭证 → 收到 200 + 数据
  → 简单，像发一个 HTTP 请求
```

已跨链处理超过 **1 亿笔交易**（截至 2026 年初），仅 Solana 链上就超过 3500 万笔。

### 11. TAP —— Trusted Agentic Commerce Protocol（可信代理商务协议）

| 属性 | 信息 |
|------|------|
| **开发者** | Forter（反欺诈公司） |
| **发布时间** | 2025 年 8 月 |
| **解决场景** | 商户侧风控：识别 AI 代理身份、防欺诈 |
| **核心创新** | 基于 Forter 的 20 亿+ 身份网络做代理身份验证 |

#### 解决什么问题？

AP2 和 ACP 关注的是"用户怎么授权代理付款"，TAP 关注的是另一面——**商户怎么识别来的是不是合法代理？**

Forter 发现：ChatGPT Agent 上线后，商户网站的代理流量暴增 **700%**，但购买转化率很低。很多商户直接封杀了代理流量。原因是：

```
商户的顾虑：
  · 来的是真的 AI 代理还是爬虫？
  · 这个代理背后是真实用户还是薅羊毛机器人？
  · 我的定价策略、库存数据被代理抓走了怎么办？
  · 代理买了东西恶意退货怎么办？
```

TAP 让商户能验证代理身份，并在不同环节（浏览商品页、加入购物车、结账）做出不同的风控决策。

### 12. ERC-8004 —— AI 代理链上身份标准

| 属性 | 信息 |
|------|------|
| **开发者** | 以太坊基金会 + MetaMask + Google + Coinbase |
| **发布时间** | 2026 年 1 月（主网上线） |
| **解决场景** | AI 代理的链上身份注册和信誉管理 |
| **核心创新** | 给每个 AI 代理一个链上"身份证" |

#### 核心概念

| 组件 | 作用 |
|------|------|
| **Identity Registry** | 给每个代理分配唯一 agentId |
| **Reputation Registry** | 记录代理的信誉评分 |
| **Validation Registry** | 验证代理的身份和权限 |

已有超过 24,000 个代理身份在以太坊上注册。这是 x402 和 AP2 的身份基础层——代理要付钱或收钱，首先得有一个可验证的身份。

### 14. 支付协议横向对比

| 维度 | AP2 (Google) | ACP (Stripe+OpenAI) | x402 (Coinbase) | TAP (Forter) |
|------|-------------|--------------------|-----------------|-----------| 
| **一句话** | 用户授权 AI 买东西的证明体系 | 商户一键接入 AI 购物渠道 | 机器之间的 HTTP 微支付 | 识别代理身份的风控协议 |
| **核心问题** | 怎么证明用户真的授权了？ | 商户怎么快速接 AI 代理？ | 机器怎么自主付钱？ | 来的代理是好人还是坏人？ |
| **场景** | 人→代理→商户 | 人→代理→商户 | 代理→代理 | 商户侧风控 |
| **支付方式** | 全部 | 传统支付 (40+ 种) | 仅加密货币 (USDC) | 不处理支付本身 |
| **信任机制** | 密码学签名 (去中心化) | 平台信任 Stripe (中心化) | 链上签名 (去中心化) | 身份网络 (中心化) |
| **凭证名称** | Mandate (VDC) | SharedPaymentToken (SPT) | PaymentPayload | 身份验证信号 |
| **开源** | 是 (Apache 2.0) | 是 | 是 (Apache 2.0) | 否（商业产品） |
| **首发平台** | Gemini | ChatGPT | 任何 HTTP 服务 | Forter 客户 |
| **商户门槛** | 中等（需集成 VDC 组件） | 低（Stripe 后台点几下） | 低（加一行中间件） | 低（Forter SDK） |
| **适合谁** | 需要密码学级别安全的场景 | 已用 Stripe 的商户 | API/数据服务提供商 | 担心代理欺诈的商户 |

### 15. 它们是竞争还是互补？

**答案：更多是互补，各解决支付链条中的一个环节。**

```
一个完整的 AI 代理购物交易中，可能同时用到多个协议：

用户："Gemini，帮我找最便宜的机票"
  │
  │ ① Gemini 加载"机票比价"Skill，获取操作流程（Skills Protocol）
  │
  │ ② Gemini 用 MCP 调用航班搜索工具（MCP 协议）
  │
  │ ③ Gemini 和携程的 AI 代理对话比价（A2A 协议）
  │
  │ ④ 携程代理验证 Gemini 的身份（TAP / ERC-8004）
  │
  │ ⑤ 用户确认购买，签署授权凭证（AP2 的 Mandate 或 ACP 的 SPT）
  │
  │ ⑥ 完成支付（传统信用卡走 ACP/Stripe，加密货币走 x402）
  │
  └──→ 交易完成
```

但在"用户授权"这个环节，AP2 和 ACP 确实是**直接竞争**关系——两者解决同一个问题，但方式不同：

| 维度 | AP2 (Google) | ACP (Stripe) |
|------|-------------|-------------|
| **哲学** | "代码即法律"——密码学证明一切 | "平台即信任"——Stripe 背书一切 |
| **优点** | 跨平台、跨支付方式、不依赖单一公司 | 简单、快速、商户零改动 |
| **缺点** | 集成复杂、生态还在建设 | 依赖 Stripe、主要走传统支付 |
| **适合** | 大型企业、跨境、多代理协作 | 中小商户、快速上线 |
| **阵营** | Google + Coinbase | OpenAI + Stripe |
