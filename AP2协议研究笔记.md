## 1. Google Agent Payments Protocol (AP2)

- **来源**: [Google Cloud Blog - Announcing AP2](https://cloud.google.com/blog/products/ai-machine-learning/announcing-agents-to-payments-ap2-protocol)
- **发布日期**: 2025-09-16
- **发布方**: Google（Stavan Parikh, VP/GM Payments; Rao Surapaneni, VP/GM Business Applications Platform）

### 1.1 概述

AP2 是 Google 联合支付和科技公司开发的**开放协议**，用于在跨平台环境下安全地发起和完成 AI 代理主导的支付交易。该协议可作为 **Agent2Agent (A2A)** 协议和 **Model Context Protocol (MCP)** 的扩展，建立一个与支付方式无关的框架，让用户、商户和支付提供商能够放心地使用各类支付方式进行交易。

### 1.2 解决的核心问题

现有支付体系的基本假设是"人类直接点击购买"，AI 代理自主发起支付打破了这一假设，AP2 针对性解决三个关键问题：

| 问题 | 说明 |
|------|------|
| **授权 (Authorization)** | 证明用户授予了代理进行特定购买的权限 |
| **真实性 (Authenticity)** | 确保代理的请求准确反映用户的真实意图 |
| **问责 (Accountability)** | 当欺诈或错误交易发生时，明确责任归属 |

### 1.3 核心机制 —— Mandate（授权凭证）

#### 一句话理解

> 传统支付：你亲自走进商店，挑好商品，亲手刷卡 —— 商家看到你本人，所以信任这笔交易。
> AI 代理支付：你派了一个"助理"去商店帮你买东西 —— 商家怎么确认这个助理真的是你派来的、真的按你的意思买的？
> **Mandate 就是你给这个助理的"授权书"**，上面有你的签名、明确的购买规则，任何人都可以验证它的真实性，且无法伪造或篡改。

#### 什么是 Mandate？

Mandate 是一份**加密签名的数字合约**，具备以下特性：

- **防篡改**：一旦签署，内容无法被任何人（包括 AI 代理自己）修改
- **可验证**：商家、支付机构等任何参与方都能独立验证其真实性
- **可追溯**：形成完整的证据链，出了问题可以回溯追责

技术上，Mandate 由 **Verifiable Credentials (VCs，可验证凭证)** 签名。可以类比为：传统世界中经过公证的委托书，但在数字世界中通过密码学实现，更快、更安全。

#### 两种 Mandate 类型

| 类型 | 用途 | 类比 |
|------|------|------|
| **Intent Mandate（意图授权）** | 记录用户"想买什么"的意图和规则 | 你写给助理的购物清单 + 预算限制 |
| **Cart Mandate（购物车授权）** | 锁定最终要买的具体商品和价格 | 你在订单上签字确认"就买这些" |

#### 场景一：实时购买（你在旁边盯着）

就像你在手机上让 AI 助手帮你买东西，你全程参与：

```
第1步：你说"帮我找一双白色跑鞋"
       → 系统生成【Intent Mandate】记录你的意图
       → 这份记录有你的加密签名，证明确实是你提出的请求

第2步：AI 代理去各个商家比价，找到合适的鞋子，呈现给你
       → 你看到：Nike Air Max, 白色, ¥899

第3步：你确认"就买这双"
       → 系统生成【Cart Mandate】锁定具体商品和价格
       → 这份记录同样有你的加密签名
       → 此时商品和价格被"冻结"，不能被偷偷更改

第4步：Cart Mandate 关联你的支付方式（信用卡/支付宝等）→ 完成付款
```

**关键点**：每一步都有你的签名确认，商家能验证"确实是用户本人授权买的这双鞋"。

#### 场景二：委托任务（你不在，代理自主行动）

就像你提前告诉 AI 助手一个任务，然后去忙别的：

```
第1步：你说"帮我抢周杰伦演唱会门票，最多花 ¥2000，只要内场"
       → 系统生成【Intent Mandate】，内容包含：
         · 商品：周杰伦演唱会门票
         · 价格上限：¥2000
         · 条件：仅限内场座位
         · 时间窗口：开售后 30 分钟内
       → 你用加密签名确认这份授权

第2步：AI 代理持续监控票务平台...

第3步：开售！代理发现内场票 ¥1800，符合所有条件
       → 代理依据 Intent Mandate 的规则，自动生成【Cart Mandate】
       → Cart Mandate 里写明：1张内场票, ¥1800
       → 因为完全符合你预设的规则，代理有权自动执行

第4步：Cart Mandate 关联支付方式 → 自动完成付款
```

**关键点**：代理不是"随便买"，而是严格按照你预设的规则执行。如果票价超过 ¥2000 或者不是内场，代理**无权**生成 Cart Mandate，交易不会发生。

#### 信任链条（为什么这套机制能防止出问题）

```
 用户签名的 Intent Mandate          用户/代理签名的 Cart Mandate           支付凭证
 ┌─────────────────────┐          ┌─────────────────────────┐          ┌──────────┐
 │ "我想买什么"         │ ──────→ │ "确认买这个，这个价格"    │ ──────→ │ 扣款记录  │
 │ + 规则/预算/条件     │          │ + 具体商品和金额         │          │          │
 │ + 用户加密签名       │          │ + 加密签名               │          │          │
 └─────────────────────┘          └─────────────────────────┘          └──────────┘
         ↑                                  ↑                               ↑
    证明用户授权了               证明买的东西符合用户意图            证明钱付对了
```

这三个环节串在一起，形成一条**不可否认的审计链**：
- **出了纠纷**：调出 Mandate 链条，一目了然是用户的意图、代理的执行还是商家的问题
- **防止代理越权**：代理只能在 Intent Mandate 规定的范围内操作
- **防止篡改**：所有环节都有加密签名，改一个字节都会导致验证失败

### 1.4 Mandate 的技术本质与存储方式

> **常见疑问**：Mandate 存在区块链上吗？

#### 结论：不在区块链上

AP2 官方将 Mandate 称为 **Verifiable Digital Credentials (VDCs，可验证数字凭证)**。它本质上是一段**加密签名的结构化数据**（JSON 格式），而非区块链上的记录。

#### 它到底是什么？

可以把 VDC 想象成一份**带有防伪印章的电子文档**：

```
┌─────────────────────────────────────────────┐
│          一个 Cart Mandate (VDC) 的内容        │
├─────────────────────────────────────────────┤
│  · 买家身份标识（加密的用户 ID）               │
│  · 卖家身份标识（商户 ID）                     │
│  · 交易详情（商品、数量、金额、币种）            │
│  · 支付方式的 Token 化表示（不暴露卡号）         │
│  · 风险相关信号（设备信息、位置等）              │
│  · 时间戳                                     │
│  ─────────────────────────────────────────── │
│  🔏 用户的加密签名（用户私钥生成）              │
│  🔏 商户的加密签名                             │
└─────────────────────────────────────────────┘
```

这里的"加密签名"使用的是**非对称加密**（公钥/私钥体系），和区块链上的签名技术同源，但不依赖区块链网络。

#### 存储在哪？谁持有？

AP2 采用**角色分离架构**，不同的 VDC 由不同的参与方持有和传递：

| 角色 | 说明 | 持有的 VDC |
|------|------|-----------|
| **User Agent (UA)** | 用户的 AI 代理（如 Gemini、ChatGPT） | Intent Mandate |
| **Credentials Provider (CP)** | 管理用户支付凭证的实体（如数字钱包） | 支付凭证的 Token |
| **Merchant Endpoint (ME)** | 商户的代理/接口 | Cart Mandate（由商户生成，用户签名） |
| **Merchant Payment Processor (MPP)** | 商户的支付处理方 | Payment Mandate → 发给支付网络 |
| **Network & Issuer** | 支付网络和发卡行（Visa/Mastercard等） | Payment Mandate（用于风控判断） |

流转过程：

```
用户设备                商户代理              支付处理方           支付网络
  │                      │                    │                   │
  │  ① 用户签署          │                    │                   │
  │  Intent Mandate      │                    │                   │
  │  (存在用户设备/代理)   │                    │                   │
  │                      │                    │                   │
  │ ──── 购物意图 ──────→ │                    │                   │
  │                      │                    │                   │
  │ ←── 生成购物车 ────── │                    │                   │
  │                      │                    │                   │
  │  ② 用户签署          │                    │                   │
  │  Cart Mandate        │                    │                   │
  │ ──── 签名后传给商户 →  │                    │                   │
  │                      │                    │                   │
  │                      │ ── Cart Mandate ──→ │                   │
  │                      │                    │                   │
  │                      │                    │ ③ 生成             │
  │                      │                    │ Payment Mandate    │
  │                      │                    │ ── 发送给网络 ────→ │
  │                      │                    │                   │
  │                      │                    │                   │ ④ 风控验证
  │                      │                    │                   │ 批准/拒绝
```

#### 为什么不用区块链？

| 维度 | 区块链方案 | AP2 的 VDC 方案 |
|------|-----------|----------------|
| **速度** | 上链确认需要数秒到数分钟 | 直接在参与方之间传递，毫秒级 |
| **成本** | 每笔交易需要 Gas 费 | 无链上成本 |
| **隐私** | 链上数据公开可查 | 数据仅在必要的参与方之间共享 |
| **兼容性** | 需要所有参与方接入同一条链 | 兼容现有支付体系（Visa/Mastercard等） |
| **防篡改** | 靠区块链共识机制 | 靠密码学签名（同样不可篡改） |

简单说：AP2 借用了区块链的**密码学思想**（加密签名、不可篡改），但没有用区块链的**基础设施**（分布式账本、共识机制）。这是一个务实的选择——让协议能无缝嵌入现有的 Visa/Mastercard 支付网络，而不需要商家和银行去跑区块链节点。

> **例外**：当支付方式本身是加密货币/稳定币时（通过 x402 扩展），结算环节会涉及链上操作，但 Mandate 本身的授权和验证仍然是链下的。

### 1.5 支持的支付方式

AP2 设计为通用协议，支持多种支付类型：

- 信用卡 / 借记卡
- 稳定币 (Stablecoins)
- 加密货币 (Cryptocurrencies)
- 实时银行转账

其中，Google 与 Coinbase、以太坊基金会、MetaMask 等合作推出了 **A2A x402 扩展**，专门支持基于加密货币的代理支付。

### 1.6 应用场景示例

|------|------|
| **智能购物** | 用户想要某款绿色夹克但缺货，代理持续监控库存和价格，在条件满足时自动购买 |
| **个性化推荐** | 用户告知代理想买自行车用于旅行，商家代理据此生成包含头盔、行李架的限时捆绑优惠 |
| **协调任务** | 用户设定预算 $700 预订周末旅行，代理同时与航空公司和酒店代理交互完成组合预订 |

## 2. A2A x402 Extension —— AI 代理之间的加密货币支付协议

- **来源**: [GitHub - google-agentic-commerce/a2a-x402](https://github.com/google-agentic-commerce/a2a-x402)
- **底层协议**: [x402 Protocol](https://x402.gitbook.io/x402)（由 Coinbase 和 Cloudflare 开发）
- **版本**: v0.1（2025-09-16 发布）
- **许可**: Apache-2.0

### 2.1 一句话理解

> AP2 解决的是"用户让 AI 代理帮自己去商家那里买东西"的支付问题。
> **A2A x402 解决的是另一个问题："AI 代理 A 请 AI 代理 B 帮忙干活，A 怎么付钱给 B？"**
>
> 就像你（用户）不需要参与，两个 AI 代理自己谈好价格、自己用加密货币结算。

### 2.2 它解决什么问题？

在 AI 代理生态中，一个代理可能需要调用另一个代理的能力（比如生成图片、翻译文本、分析数据）。问题是：**这些服务怎么收费？**

传统方式：注册账号 → 申请 API Key → 绑定信用卡 → 按月结算。这对 AI 代理来说太复杂了。

x402 的方式：**即用即付，无需注册，加密货币实时结算**。

### 2.3 x402 底层协议

A2A x402 是基于 **x402 协议** 构建的。x402 协议复活了 HTTP 状态码中从未被广泛使用的 **402 Payment Required**。

核心思想非常简单：

```
传统 HTTP 请求：
  客户端 → 请求资源 → 服务器返回 200 OK + 数据

x402 HTTP 请求：
  客户端 → 请求资源 → 服务器返回 402 Payment Required（告诉你要付多少钱）
  客户端 → 签名支付 → 重新请求（附带支付凭证）
  服务器 → 验证支付 → 链上结算 → 返回 200 OK + 数据
```

| 特性 | 说明 |
|------|------|
| **结算速度** | 约 2 秒 |
| **最小金额** | 低至 $0.01（适合微支付） |
| **默认代币** | USDC（美元稳定币） |
| **支持的链** | Base、Ethereum、Polygon 等 |
| **平台费用** | 零平台费，商户直收 |

### 2.4 A2A x402 的完整支付流程

在 A2A（Agent-to-Agent）协议中，x402 定义了四个角色：

| 角色 | 说明 | 类比 |
|------|------|------|
| **Client Agent** | 需要服务的 AI 代理 | 雇主 |
| **Merchant Agent** | 提供付费服务的 AI 代理 | 服务商 |
| **Signing Service** | 管理钱包私钥、负责签名 | 出纳/财务 |
| **Facilitator** | 验证支付并在链上结算 | 银行/清算所 |

完整交易流程（以"AI 代理 A 请 AI 代理 B 生成一张图片"为例）：

```
  Client Agent (A)                    Merchant Agent (B)               区块链
       │                                     │                          │
  ① A: "帮我生成一张猫的图片"                  │                          │
       │ ─── 发送请求 ──────────────────────→ │                          │
       │                                     │                          │
       │    B: "可以，但要付 0.05 USDC"        │                          │
       │ ←── 返回 402 + 支付要求 ──────────── │                          │
       │                                     │                          │
       │   支付要求包含：                       │                          │
       │   · 金额：0.05 USDC                  │                          │
       │   · 收款地址：0xB的钱包                │                          │
       │   · 支持的链：Base                    │                          │
       │   · 有效期：600秒                     │                          │
       │                                     │                          │
  ② A 决定接受，用钱包签名支付授权               │                          │
       │ ─── 发送签名后的支付凭证 ───────────→ │                          │
       │                                     │                          │
       │                               ③ B 验证签名                     │
       │                                  链上结算 ─────────────────────→│ 转账
       │                                     │                          │ 0.05 USDC
       │                                     │                          │ A → B
       │                               ④ B 开始生成图片                  │
       │                                     │                          │
       │    B: "搞定了，这是你的图片"           │                          │
       │ ←── 返回结果 + 链上交易凭证 ───────── │                          │
       │                                     │                          │
       │   凭证包含：                          │                          │
       │   · 交易哈希：0xabc123...             │                          │
       │   · 网络：Base                       │                          │
       │   · 状态：成功                        │                          │
```

### 2.5 支付状态机

交易有 6 个状态，流转清晰：

```
                    ┌──────────────────────┐
                    │   payment-required   │  商户发出支付要求
                    └──────────┬───────────┘
                               │
                ┌──────────────┼──────────────┐
                ↓                             ↓
   ┌────────────────────┐        ┌───────────────────┐
   │  payment-rejected  │        │ payment-submitted  │  客户提交签名支付
   └────────────────────┘        └─────────┬─────────┘
         (客户拒绝)                         ↓
                              ┌─────────────────────┐
                              │  payment-verified    │  商户验证签名通过
                              └─────────┬───────────┘
                                        │
                         ┌──────────────┼──────────────┐
                         ↓                             ↓
              ┌─────────────────┐          ┌──────────────────┐
              │ payment-completed│          │  payment-failed   │
              └─────────────────┘          └──────────────────┘
                (链上结算成功)                (结算失败)
```

### 2.6 安全设计（含原版 x402 与 A2A x402 的密钥管理对比）

| 安全措施 | 说明 |
|---------|------|
| **私钥隔离** | 私钥绝对不能交给 LLM 处理，必须由独立的签名服务管理 |
| **签名验证** | 商户必须在结算前密码学验证每一笔支付签名 |
| **防重放攻击** | 服务端必须追踪已使用的 nonce，防止同一笔支付被重复提交 |
| **传输加密** | 所有通信必须使用 HTTPS/TLS |

#### 原版 x402 的密钥安全方案

原版 x402 协议本身**不规定密钥如何存储**，只给了一条原则和一个推荐方案：

**原则**：买方在本地签名，卖方只验证签名、永远不持有买方的私钥。

**推荐方案**：使用 Coinbase 的 **CDP Wallet API**（Server Wallet v2）来管理密钥。

CDP Wallet API 底层使用 **AWS Nitro Enclaves（可信执行环境 TEE）**，核心安全设计如下：

```
┌─────────────────────────────────────────────────────┐
│              AWS Nitro Enclave (TEE)                 │
│                                                     │
│  · 私钥在 TEE 内部生成                                │
│  · 私钥在 TEE 内部加密                                │
│  · 签名操作在 TEE 内部完成                             │
│  · 私钥永远不离开这个隔离环境                           │
│                                                     │
│  特性：                                              │
│  · 无持久化存储（关机即清除）                           │
│  · 无交互式访问（没人能"登录"进去看）                   │
│  · 无外部网络连接（只接受签名请求，不主动联网）           │
│                                                     │
│  加密后的密钥版本存储在 CDP 数据库中                    │
│  → 必须配合 Wallet Secret 才能解密使用                  │
│  → Wallet Secret 使用 ECDSA secp256r1 椭圆曲线认证     │
└─────────────────────────────────────────────────────┘
```

用生活类比：TEE 就像一个**透明的保险柜**——你可以把需要签名的文件递进去，里面的机器人帮你签好再递出来，但没人能打开保险柜、拿走里面的印章。

**但原版 x402 有一个关键缺陷**：它只是**建议**你用安全钱包，并没有在协议层面做强制分离。也就是说：

```
原版 x402 允许这种写法（不安全但协议不阻止）：

  const wallet = new Wallet(privateKey);  // 私钥直接写在代码里
  const signature = wallet.sign(payload);  // 程序直接签名
```

对于普通的程序调 API 场景，这个风险还可控（开发者自己负责）。
但对于 AI 代理场景，就很危险了——LLM 可能被 prompt injection 攻击，诱导它泄露或滥用私钥。

#### Google A2A x402 的安全升级

Google 的 A2A x402 Extension 在协议层面**强制要求**架构分离：

```
原版 x402（建议但不强制）：
  ┌───────────────────────┐
  │     Client 程序        │
  │  · 业务逻辑            │
  │  · 钱包/私钥管理  ←危险 │
  │  · 签名操作            │
  └───────────────────────┘

A2A x402（协议层面强制）：
  ┌───────────────────┐          ┌───────────────────┐
  │  Client Agent      │          │  Signing Service   │
  │  (LLM 驱动)        │ ──请求──→│  (独立服务)         │
  │                    │          │                    │
  │  · 业务逻辑         │          │  · 管理私钥         │
  │  · 决定是否付款      │←─签名──  │  · 执行签名         │
  │  · ❌ 碰不到私钥    │          │  · 验证权限         │
  └───────────────────┘          └───────────────────┘
        ↑                               ↑
   LLM 可能被攻击               私钥在这里，与 LLM 物理隔离
   但拿不到私钥
```

| 对比 | 原版 x402 | A2A x402 Extension |
|------|-----------|-------------------|
| 密钥存储规范 | 协议不规定，建议用 CDP Wallet API | 协议强制要求与 LLM 分离 |
| 安全级别 | 取决于开发者实现 | 架构层面保证 |
| 推荐方案 | Coinbase CDP (TEE) | 独立 Signing Service |
| 防 prompt injection | 未考虑（设计时没有 LLM 场景） | 核心设计目标 |
| 适用场景 | 普通程序/脚本调用 API | AI 代理（LLM 驱动）|

### 2.7 与 AP2 的关系

| 维度 | AP2 | A2A x402 |
|------|-----|----------|
| **场景** | 用户 → AI 代理 → 商家（帮用户买东西） | AI 代理 A → AI 代理 B（代理之间买服务） |
| **支付方式** | 信用卡、借记卡、稳定币等（全支持） | 仅加密货币（USDC 为主） |
| **用户参与** | 需要用户签名授权 | 代理自主完成，用户可能不知情 |
| **结算方式** | 走传统支付网络（Visa/Mastercard） | 链上结算（Base/Ethereum/Polygon） |
| **典型金额** | 日常购物（几十到几千元） | 微支付（$0.01 起） |
| **关联** | AP2 通过 x402 扩展支持加密货币支付 | x402 是 AP2 在加密支付场景的具体实现 |

简单说：**AP2 是面向消费者购物的协议，A2A x402 是面向代理之间服务交易的协议**，两者通过 x402 扩展打通。

### 2.8 Google 的 A2A x402 到底在原版 x402 上扩展了什么？

#### 先理解原版 x402（Coinbase 开发）

原版 x402 是一个**非常简洁的 HTTP 支付协议**，设计理念是"像访问网页一样付款"：

```
原版 x402 流程（纯 HTTP）：

客户端 ──── GET /api/generate-image ──────────────→ 服务器
客户端 ←─── 402 + PAYMENT-REQUIRED 头（JSON）──── 服务器     "要付 0.05 USDC"
客户端 ──── GET /api/generate-image ──────────────→ 服务器     重发请求
             + PAYMENT-SIGNATURE 头（签名后的支付）
客户端 ←─── 200 OK + 图片数据 ────────────────── 服务器
             + PAYMENT-RESPONSE 头（链上交易凭证）
```

核心特点：**无状态、一问一答、通过 HTTP 头传递支付信息、没有"代理"的概念**。

#### Google 扩展了什么？（6 大增强）

**增强 1：从"无状态 HTTP"升级为"有状态任务"**

| 原版 x402 | A2A x402 Extension |
|-----------|-------------------|
| 每次请求独立，无状态 | 引入 `taskId` 关联整个交易生命周期 |
| 支付和服务是一次 HTTP 往返 | 支付和服务可以是多步异步流程 |

原版 x402 适合简单的"付钱→拿结果"场景。但 AI 代理的服务可能很复杂（比如生成视频需要 10 分钟），需要跟踪状态。Google 用 A2A 的 **Task 对象** 解决了这个问题——一个 taskId 串起整个"请求→付款→处理中→完成"的过程。

**增强 2：定义了明确的支付状态机**

原版 x402 没有定义支付状态，只有 HTTP 状态码（402 / 200）。Google 定义了 **6 个显式状态**：

```
payment-required → payment-submitted → payment-verified → payment-completed
                 ↘ payment-rejected                     ↘ payment-failed
```

每个状态都写在消息的 metadata 里，任何参与方都能精确知道"当前这笔交易到哪一步了"。

**增强 3：支付信息从 HTTP 头搬到了消息体**

| 原版 x402 | A2A x402 Extension |
|-----------|-------------------|
| 支付要求放在 HTTP `PAYMENT-REQUIRED` 头 | 支付要求放在 A2A Message 的 `metadata` 字段 |
| 支付签名放在 HTTP `PAYMENT-SIGNATURE` 头 | 支付签名放在 A2A Message 的 `metadata` 字段 |
| 结算凭证放在 HTTP `PAYMENT-RESPONSE` 头 | 结算凭证放在 A2A Message 的 `metadata` 字段 |

为什么要这样做？因为 A2A 代理之间的通信不一定是简单的 HTTP 请求-响应。它们通过 A2A 协议的 `message/send` 方法交换结构化消息，把支付信息嵌入消息体更自然、更灵活。

**增强 4：代理能力发现机制**

原版 x402：客户端必须**事先知道**某个 URL 需要付费，去请求后才收到 402。

A2A x402：商户代理通过 **AgentCard**（代理名片）提前声明"我支持 x402 支付"：

```json
{
  "capabilities": {
    "extensions": [{
      "uri": "https://github.com/google-a2a/a2a-x402/v0.1",
      "description": "支持 x402 链上支付",
      "required": true
    }]
  }
}
```

客户端代理在决定调用某个服务之前，就能通过 AgentCard 知道"这个代理要收费、用什么方式收费"，而不是盲目请求后才发现要付钱。

**增强 5：强制分离签名服务（安全增强）**

原版 x402：只说"客户端用钱包签名"，没有规定钱包和客户端的关系。

A2A x402 **明确要求**：签名私钥必须由独立的 **Signing Service** 管理，**绝对不能交给 LLM**。

```
原版 x402：
  客户端（含钱包）→ 签名 → 发送

A2A x402：
  Client Agent (LLM) → 请求签名 → Signing Service (独立服务，管理私钥) → 返回签名
                       ↑ LLM 永远碰不到私钥
```

这是因为 LLM 可能被 prompt injection 攻击，如果 LLM 能直接操作私钥，攻击者可以诱导它转走所有资金。

**增强 6：交易凭证持久化记录**

原版 x402：结算凭证在 HTTP 响应头返回一次就结束了。

A2A x402：引入 `x402.payment.receipts` —— 一个**持久化数组**，记录该 Task 生命周期内所有的结算尝试（包括失败的）：

```json
"x402.payment.receipts": [
  { "success": false, "errorReason": "insufficient_funds", "network": "base" },
  { "success": true, "transaction": "0xabc123...", "network": "base", "payer": "0x..." }
]
```

这意味着如果第一次付款失败（余额不足），代理换个钱包重试成功，整个过程都有完整记录，方便对账和审计。

#### 一图总结

```
┌──────────────────────────────────────────────────────────────┐
│                    原版 x402 (Coinbase)                       │
│                                                              │
│  · 纯 HTTP 协议（402 状态码 + HTTP 头）                        │
│  · 无状态，一问一答                                            │
│  · 没有代理/任务概念                                           │
│  · 客户端自己管钱包                                            │
│  · 单次凭证返回                                               │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│              Google A2A x402 Extension 新增                   │
│                                                              │
│  ✦ Task 状态管理（taskId 关联完整交易生命周期）                   │
│  ✦ 6 状态支付状态机（精确追踪每一步）                            │
│  ✦ 支付信息嵌入 A2A Message metadata（更结构化）                │
│  ✦ AgentCard 能力发现（提前知道要付费）                         │
│  ✦ 强制签名服务分离（LLM 不碰私钥）                            │
│  ✦ 交易凭证持久化数组（完整审计记录）                            │
│  ✦ 7 种标准错误码（统一错误处理）                               │
│                                                              │
│  本质：把 x402 的支付能力"嵌入"到 A2A 代理通信框架中             │
└──────────────────────────────────────────────────────────────┘
```

简单说：**原版 x402 是给"程序调用 API"设计的支付协议，Google 把它改造成了给"AI 代理之间协作"设计的支付协议**。核心支付逻辑没变（还是用加密货币链上结算），但套上了 A2A 的代理通信框架，加了状态管理、能力发现、安全隔离和审计追踪。

## 3. 战略分析：Google 和 Coinbase 在协议中的角色与商业卡位

### 3.1 先看 AP2 的角色体系

AP2 定义了 6 个角色，每个角色都是一个"卡位机会"：

```
                    ┌──────────┐
                    │   用户    │  真人，发起购物意图
                    └────┬─────┘
                         │
                    ┌────▼─────┐

                    └────┬─────┘
                         │
          ┌──────────────┼──────────────┐
          │              │              │
   ┌──────▼──────┐ ┌────▼─────┐ ┌──────▼──────────┐
   │ Credentials  │ │ Merchant │ │ Merchant Payment│
   │  Provider    │ │ Endpoint │ │   Processor     │
   │ (管理支付凭证)│ │(商户代理) │ │  (支付处理方)    │
   └──────────────┘ └──────────┘ └────────┬────────┘
                                          │
                                   ┌──────▼──────┐
                                   │ Network &   │
                                   │   Issuer    │
                                   │(支付网络/发卡行)│
```

```
  │
  └─→ 整个流程遵循 AP2 协议
        ① 协议 = Google 设计的
```

**Google 的战略意图**：通过制定开放协议，把自己的产品（Gemini、Google Pay、Google Cloud）嵌入到 AI 代理购物的每一个关键环节。协议是开放的，但 Google 在每个角色上都有先发优势。

### 3.6 关键洞察

1. **Google 是"协议定义者 + 入口控制者"**：通过制定 AP2/A2A/MCP 三大协议，定义了 AI 代理经济的游戏规则，同时通过 Gemini 和 Google Pay 控制用户入口和支付凭证。赚的是**流量钱和基础设施钱**。

2. **Coinbase 是"加密支付基础设施垄断者"**：在 x402 体系中，Facilitator（清算）、Wallet（密钥）、Exchange（入金）三个环节全部由 Coinbase 占据。虽然协议是开放的（任何人可以做 Facilitator），但目前只有 Coinbase 有生产级的实现。赚的是**每笔交易的微利 + 交易所手续费**。

3. **"开放协议"是战略武器**：两家都把协议做成开放标准——不是因为慈善，而是为了让整个行业采用自己的标准。一旦标准被广泛采用，在标准中占据关键角色的公司就成了"基础设施"，很难被替代。这和 Google 做 Android 开源、Coinbase 做 Base 链的逻辑一样。

4. **当前是"蜜月期"**：Google 需要 Coinbase 的加密支付能力，Coinbase 需要 Google 的用户和流量。但随着 AI 代理支付规模扩大，双方在交汇地带（加密钱包、清算服务）的竞争不可避免。

## 4. Stripe 的入场方式：Agentic Commerce Protocol (ACP) + 商业套件

- **来源**: [Stripe Blog - Agentic Commerce Suite](https://stripe.com/blog/agentic-commerce-suite)
- **协议**: [Agentic Commerce Protocol (ACP)](https://www.agenticcommerce.dev/)
- **合作方**: OpenAI（首发平台是 ChatGPT）

### 4.1 一句话理解

> Google 的思路是"我来定义 AI 代理支付的标准协议，大家都用我的协议"。
> **Stripe 的思路是"商户已经在用我了，我帮你一键接入 AI 代理渠道，就像多加一个销售渠道"。**
>
> Google 是从协议层往上建生态，Stripe 是从商户层往下接协议。

### 4.2 核心产品：Agentic Commerce Suite

Stripe 的产品定位非常直接——帮助现有商户"一键接入" AI 代理购物：

```
商户现在的问题：

"我想让 ChatGPT/Gemini 这些 AI 代理能卖我的东西，
 但每接一个代理平台就要花 6 个月做对接，太累了。"

Stripe 的方案：

"你的商品目录接到 Stripe，在后台勾选想接入的 AI 代理平台，
 剩下的事（发现、结账、支付、风控）我全帮你搞定。"
```

Agentic Commerce Suite 包含三个核心能力：

| 能力 | 说明 |
|------|------|
| **商品发现** | Stripe 托管 ACP 端点，AI 代理可以搜到你的商品（价格、库存、描述实时同步） |
| **结账流程** | 基于 Stripe Checkout Sessions API，处理税费、运费、库存等，商户保持控制权 |
| **支付风控** | 处理 SharedPaymentToken，配合 Stripe Radar 做代理场景的反欺诈 |

首批接入品牌：Etsy、Urban Outfitters、Coach、Kate Spade、Ashley Furniture、Revolve 等。

### 4.3 核心创新：SharedPaymentToken (SPT)

SPT 是 Stripe 在 ACP 协议中发明的**支付凭证**，相当于 AP2 的 Mandate 在 Stripe 体系中的对应物：

```
传统支付：
  用户把信用卡号给商户 → 商户直接扣款

SPT 支付：
  用户的信用卡已存在 AI 代理平台（如 ChatGPT）
  → 代理生成一个 SPT（不含真实卡号）
  → SPT 发给商户
  → 商户用 SPT 通过 Stripe 完成扣款

  SPT 的限制：
  · 绑定特定商户（不能拿去别的商户用）
  · 限定金额上限
  · 限定有效期
  · 全程可追踪
```

| 对比 | AP2 的 Mandate (VDC) | Stripe 的 SPT |
|------|---------------------|---------------|
| **本质** | 加密签名的意图证明 | 受限的支付 Token |
| **保护什么** | 用户意图不被篡改 | 支付凭证不被滥用 |
| **谁签名** | 用户自己用私钥签 | AI 代理平台生成 |
| **适用范围** | 跨平台、跨支付方式 | Stripe 生态内 |
| **技术基础** | 密码学 + 可验证凭证 | Stripe API + Token 化 |

## 5. Circle 的入场方式：USDC + 可编程钱包 + x402

- **来源**: [Circle Blog - Autonomous Payments](https://www.circle.com/blog/autonomous-payments-using-circle-wallets-usdc-and-x402)
- **核心产品**: Developer-Controlled Wallets + USDC + x402 集成
- **开发者文档**: [Circle Docs](https://developers.circle.com/)

### 5.1 一句话理解

> Google 做协议标准，Stripe 做商户工具，Coinbase 做清算基础设施。
> **Circle 的思路是"让 USDC 成为 AI 代理世界的默认货币"。**
>
> 不管你用什么协议、什么平台，只要代理之间要转钱，就用 USDC。

### 5.2 Circle 是谁？

Circle 是 **USDC（美元稳定币）的发行方**。USDC 是目前市值第二大的稳定币，1 USDC = 1 美元，由美元储备 1:1 支撑。Circle 的核心商业模式：

```
用户/企业 → 存入美元 → Circle 发行等量 USDC
                     → Circle 把美元存入国债/银行
                     → 靠储备金利息赚钱（年化数十亿美元）

流通的 USDC 越多 → Circle 赚的利息越多
```

**所以 Circle 的核心利益是：让 USDC 在尽可能多的场景中流通。AI 代理支付是一个巨大的增量场景。**

### 5.3 核心产品组合

Circle 为 AI 代理支付提供了三层产品：

```
┌──────────────────────────────────────────────┐
│  第三层：协议集成                               │
│  · x402 协议支持（HTTP 原生支付）               │
│  · AP2 协议合作（Google 生态）                  │
│  · MCP Server（开发者 IDE 集成）               │
├──────────────────────────────────────────────┤
│  第二层：钱包即服务（WaaS）                     │
│  · Developer-Controlled Wallets               │
│  · MPC 密钥管理（私钥不暴露）                   │
│  · 多链支持（EVM + Solana）                    │
│  · EOA 和智能合约账户                           │
├──────────────────────────────────────────────┤
│  第一层：货币层                                │
│  · USDC 发行和赎回                             │
│  · 跨链转账（CCTP 协议）                        │
│  · Testnet 水龙头（开发测试用）                  │
└──────────────────────────────────────────────┘
```

#### Developer-Controlled Wallets（核心产品）

Circle 的钱包方案与 Coinbase CDP 的定位不同：

| 维度 | Coinbase CDP Wallet | Circle Developer-Controlled Wallets |
|------|--------------------|------------------------------------|
| **托管模式** | TEE（可信执行环境） | MPC（多方计算） |
| **控制权** | 开发者通过 API 调用 | 开发者完全控制钱包操作 |
| **定位** | x402 生态的默认钱包 | 通用的"钱包即服务" |
| **特点** | 跟 Coinbase 交易所打通 | 跟 USDC 发行/赎回打通 |

MPC（多方计算）的原理用一句话解释：**把一把钥匙拆成多片，分别存在不同地方，签名时各片协作计算，但完整钥匙从不在任何一个地方出现过。**

### 5.4 Circle 的 AI 代理支付场景

Circle 已经实现了一个完整的 Demo（使用 LangChain + GPT-4o mini）：

```
AI 代理想查询某个钱包的风险评分（一个付费 API）

① 代理调用 Circle API 创建钱包
② 代理调用 Circle 水龙头 API 给钱包充值 USDC
③ 代理请求风险评分 API → 收到 402 (需支付 $0.01)
④ 代理调用 Circle 签名 API，签署支付授权
⑤ 代理重发请求 + 支付签名 → 服务端验证 → 链上结算
⑥ 代理收到风险评分数据

全程无人工干预，代理自己创建钱包、充值、付费、拿数据。
```

### 5.5 Circle 的战略逻辑

**入场方式：从"货币层"切入，"卖水"策略**

```
淘金热中谁最赚钱？不是挖金子的人，是卖水的。

AI 代理支付的"淘金热"中：
  · Google 在卖"矿区地图"（协议标准）
  · Stripe 在卖"挖矿工具"（商户接入套件）
  · Coinbase 在卖"淘金铲子"（清算基础设施）
  · Circle 在卖"水"（USDC，代理之间流通的货币本身）

不管谁的协议赢了、谁的工具好用，
只要代理之间用 USDC 交易，Circle 就赚钱。
```

Circle 的策略是**与所有协议兼容**，而不是自己做协议：

| 协议生态 | Circle 的角色 |
|---------|-------------|
| **x402 (Coinbase)** | USDC 是默认支付代币 + Circle Wallet 是替代钱包方案 |
| **AP2 (Google)** | AP2 的 x402 扩展使用 USDC 结算 |
| **ACP (Stripe/OpenAI)** | 未来可作为加密支付选项 |
| **通用场景** | 任何 AI 代理的钱包都可以持有和转移 USDC |

### 6.5 关键洞察

   - **Google + Coinbase 阵营**：AP2 + A2A + x402，偏向开放协议 + 加密支付，更适合复杂的多代理协作场景

4. **最终可能不是零和博弈**：就像线上支付领域 Visa、Mastercard、PayPal、支付宝共存一样，AI 代理支付也可能形成多协议共存的局面——简单购物用 ACP/Stripe，复杂协作用 AP2/Google，代理间微支付用 x402/Coinbase，而 USDC 在所有加密场景中通用。

```
│     · 发行虚拟/实体预付卡，余额由稳定币支撑                     │
│     · 持卡人可在任何接受 Visa 的地方消费                       │
│     · 稳定币余额自动转换为法币完成刷卡                          │
│     · 私有预览阶段                                            │
│                                                             │
│  ④ 开放发行平台（面向发币方）                                  │
│     · 任何企业几行代码就能发行自己的稳定币                      │
│     · Bridge 处理储备管理、合规、流动性                         │
│     · 储备由 BlackRock + Fidelity 管理                        │
│     · 首批案例：Phantom 的 CASH、MetaMask 的 mUSD             │
│                                                             │
│  ⑤ 法币→加密入金通道（面向开发者）                              │
│     · Stripe 托管的入金页面 或 嵌入式组件                      │
│     · 用户用信用卡/银行转账买加密货币                           │
│     · 支持 iOS/Android/React Native                          │
└─────────────────────────────────────────────────────────────┘
```

#### 产品逻辑

Stripe 的加密货币策略**不是"让大家炒币"，而是"让稳定币变成更好的美元"**：

```
传统跨境转账：
  美国企业 → 银行 → SWIFT → 中间行 → 本地银行 → 菲律宾员工
  耗时：3-5 天 | 费用：$25-50 | 汇率损失：1-3%

Stripe 稳定币转账：
  美国企业 → Stripe 金融账户（稳定币）→ 链上转账 → 员工钱包
  耗时：秒级 | 费用：近零 | 汇率：实时透明

关键：用户不需要懂"区块链"，只需要在 Stripe 后台操作
```

**USDB 是 Stripe 的秘密武器**：Bridge 发行的 USDB 是"闭环稳定币"，与 USDC 不同——USDB 的储备金利息归 Stripe/Bridge，而不是像 USDC 那样归 Circle。这意味着 Stripe 正在构建自己的"货币层"，逐步减少对 Circle 的依赖。

### 7.2 智能体商务产品线（Agentic Commerce）

这是 Stripe 面向"AI 代理帮用户购物"场景的产品线，核心是让 AI 代理平台（ChatGPT 等）和商户之间能够完成交易。

#### 产品架构

```
┌──────────────────────────────────────────────────────────────┐
│               Stripe Agentic Commerce 架构                    │
│                                                              │
│  ┌──────────┐      ┌────────────────┐      ┌──────────────┐  │
│  │  Buyer   │      │   AI Agent     │      │   Seller     │  │
│  │  (买家)   │      │  (ChatGPT等)   │      │   (商户)     │  │
│  │          │      │               │      │              │  │
│  │ 说"我要   │─────→│ 搜索商品       │─────→│ ACP 端点     │  │
│  │  买鞋"    │      │ 展示选项       │←─────│ 返回商品      │  │
│  │          │      │ 发起结账       │      │              │  │
│  │ 确认购买  │─────→│ 生成 SPT      │─────→│ 用 SPT 扣款  │  │
│  │          │      │              │      │ (通过 Stripe) │  │
│  └──────────┘      └──────┬───────┘      └──────────────┘  │
│                           │                                  │
│                    ┌──────▼───────┐                          │
│                    │Payment       │                          │
│                    │Provider      │                          │
│                    │(Stripe)      │                          │
│                    │· Token 化     │                          │
│                    │· 风控 (Radar) │                          │
│                    │· 结算         │                          │
│                    └──────────────┘                          │
└──────────────────────────────────────────────────────────────┘
```

#### 三个核心子产品

**① Agentic Commerce Protocol (ACP) —— 协议层**

ACP 定义了 AI 代理和商户之间的标准化交互接口：

| 端点 | 作用 |
|------|------|
| `CreateCheckout` | 创建结账会话（商品、数量、客户信息） |
| `UpdateCheckout` | 更新结账（改数量、地址、优惠码等） |
| `CompleteCheckout` | 完成结账（提交 SPT 和支付） |
| `CancelCheckout` | 取消结账 |

支持两种实现方式：RESTful HTTP 和 MCP Server。

**② Agentic Commerce Suite —— 商户工具**

解决的问题：商户接一个 AI 代理平台要 6 个月，Stripe 让它变成一键操作：

```
商户操作：
  1. 把商品目录接入 Stripe（上传或从现有 ERP 同步）
  2. 在 Stripe 后台勾选要接入的 AI 代理平台
  3. 完成 ✓

Stripe 自动处理：
  · 商品信息同步给各 AI 代理平台
  · 结账流程（税费、运费、库存）
  · 支付处理 + 风控
  · 订单事件通知 → 商户用自己的物流系统发货
```

已接入的电商平台：Wix、WooCommerce、BigCommerce、Squarespace、commercetools。

**③ SharedPaymentToken (SPT) —— 支付凭证**

SPT 是整个体系的安全核心，设计非常精巧：

```
SPT 的生命周期：

① 用户在 ChatGPT 上存了一张 Visa 卡

② 用户说"帮我在 Etsy 上买这个花瓶"

③ ChatGPT 通过 Stripe API 生成一个 SPT：
   {
     "id": "spt_1RgaZc...",
     "usage_limits": {
       "currency": "usd",
       "max_amount": 5000,        // 限额 $50.00
       "expires_at": 1751587220   // 30 分钟后过期
     },
     "seller_details": {
       "network_id": "etsy",      // 只能 Etsy 用
       "external_id": "shop_123"  // 只能这个店铺用
     }
     // ❌ 不包含真实卡号
     // ✓ 只含有限信息：卡品牌 + 后四位
   }

④ SPT 发给 Etsy → Etsy 用 SPT 创建 PaymentIntent → Stripe 扣款

⑤ Webhook 通知：
   · Etsy 收到 "spt.used" 事件 → 确认收款
   · ChatGPT 收到 "spt.used" 事件 → 通知用户
   · SPT 使用后自动失效，不能二次使用
```

**SPT vs AP2 Mandate 的核心区别**：

| 维度 | SPT (Stripe) | Mandate/VDC (AP2) |
|------|-------------|-------------------|
| **设计哲学** | "我来管支付安全" | "让密码学保证安全" |
| **信任根基** | Stripe 平台信任（中心化） | 加密签名验证（去中心化） |
| **卡号保护** | Token 化，Stripe 托管真实卡号 | 与卡号无关，是意图证明 |
| **适用场景** | 传统信用卡/借记卡支付 | 任何支付方式（含加密） |
| **商户门槛** | 已用 Stripe 的商户几乎零改动 | 需要深度集成密码学组件 |

### 7.3 机器支付产品线（Machine Payments）

这是 Stripe 面向"机器之间直接付钱"的产品线——没有人类参与，AI 代理或程序自主用加密货币支付。

#### 与智能体商务的区别

```
智能体商务（Agentic Commerce）：
  用户 → "Gemini，帮我买鞋" → AI 代理 → 商户
  ↑ 有人类参与：用户发起、用户确认、用户的信用卡付款

机器支付（Machine Payments）：
  AI 代理 A → 需要调用翻译 API → 直接用 USDC 付费 → 拿到翻译结果
  ↑ 无人类参与：代理自主决定、自主付款（加密货币）、自主拿结果
```

#### 产品特点

| 特性 | 说明 |
|------|------|
| **协议** | 基于 **x402 协议**（Coinbase 开发，Stripe 集成） |
| **最低金额** | 0.01 USDC（约 ¥0.07） |
| **支持链** | Base（以太坊 L2） |
| **Gas 费** | Stripe 代付，商户零成本 |
| **结算方式** | 链上结算后自动进入 Stripe 余额 |
| **报表/退款** | 和普通 Stripe 支付完全一致 |
| **隐私** | 每笔支付生成唯一存款地址，减少链上关联 |

#### 集成方式

Stripe 把 x402 集成做得极其简单——加一行中间件：

```javascript
// 给 /paid 端点加上支付门槛：每次请求收 $0.01 USDC
app.use(
  paymentMiddleware({
    "GET /paid": {
      price: "$0.01",
      network: "base",
      payTo: createPayToAddress,   // Stripe 自动生成存款地址
    }
  })
)
```

背后 Stripe 自动处理：创建 PaymentIntent → 生成链上存款地址 → 等待链上确认 → 自动 capture → 资金进入 Stripe 余额。

#### 与 Coinbase x402 的关系

Stripe 做机器支付时，做了一个出人意料的选择——**直接采用了竞争对手 Coinbase 开发的 x402 协议**：

```
为什么 Stripe 不自己做一套机器支付协议？

① x402 已经有了社区和采用量，重新发明轮子没有意义
② Stripe 的核心竞争力不在协议层，而在"商户体验"层
③ 通过兼容 x402，Stripe 能接入所有已用 x402 的 AI 代理客户端
④ Stripe 的差异化在于：
   · 链上支付自动汇入 Stripe 余额（统一报表）
   · Stripe 代付 Gas 费
   · 每笔支付独立存款地址（隐私保护）
   · 和 Stripe 现有的退款/争议/多币种结算体系打通
```

| 维度 | 直接用 Coinbase x402 | 通过 Stripe 用 x402 |
|------|---------------------|-------------------|
| 收到的钱 | 在链上钱包里（需自己管理） | 在 Stripe 余额里（和信用卡收入统一） |
| Gas 费 | 自己付或用 Coinbase Facilitator | Stripe 代付 |
| 报表 | 自己做链上对账 | Stripe Dashboard 统一查看 |
| 退款 | 自己发链上交易退 | Stripe 退款流程 |
| 合规 | 自己处理 KYT/KYC | Stripe 内置 |

```

                    │    加密货币，不管是买实体商品还是调     │
                    │    API，所有支付都走 Stripe"          │
                    └──────────┬──────────────────────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
       ┌──────▼──────┐  ┌─────▼─────┐  ┌───────▼──────┐

                    └─────────────────────┘
```

**核心逻辑**：这三条产品线不是独立的，而是一个完整的飞轮：

1. **Crypto** 让商户能接受稳定币 → 为 Machine Payments 打下基础
2. **Agentic Commerce** 让 AI 代理帮用户在商户购物 → 增加 Stripe 交易量
3. **Machine Payments** 让 AI 代理之间直接付费 → 创造全新的交易场景
4. **所有资金都汇入 Stripe Balance** → 商户不需要管理多套账本

Stripe 的底层假设：**未来的"客户"不仅是人类，还包括 AI 代理和自动化程序**。谁能同时服务好人类客户、AI 代理客户和机器客户，谁就占据了支付的未来。
